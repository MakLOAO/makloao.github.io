SpringBoot08-消息队列

消息队列广泛用于异步处理，应用解耦和流量削峰

JMS：基于JVM消息代理的规范，ActiveMQ是JMS的实现
AMQP：高级消息队列协议，兼容JMS，RabbitMQ是AMQP的实现

Spring也有对应的支持，可以查阅官方文档
它提供 JmsTemplate 和 RabbitTemplate 来发送消息
@JmsListener 和 RabbitListener 注解在方法上监听消息代理发布的消息
@EnableJms 和 @EnableRabbit 来开启支持

SpringBoot的自动配置： JmsAutoConfiguration 和 RabbitAutoConfiguration

RabbitMQ

简介

  • Message：消息，不具名，由消息头和消息体组成，消息体是不透明的，消息头是由一系列可选属性组成，包括routing-key（路由键），priority（优先权），delivery-mode（指出该消息可能需要持久性存储）等
  • Publisher：消息生产者，一个向交换器发布消息的客户端应用程序
  • Exchange：交换器，用来接收生产者发送的消息并将这些消息路由给服务器中的队列，有4种类型：direct（默认），fanout，topic和headers，不同类型转发消息策略不同
  • Queue：消息队列，用来保存消息直到发送给消费者，它是消息的容器，也是消息的终点，一个消息可投入一个或多个队列，消息会一直在队列里，等待消费者连接到这个队列将其取走
  • Binding：绑定，用于消息队列和交换器之间的关联，一个绑定就是基于路由键将交换器和消息队列连接起来的路由规则，所以可以将交换器理解成一个由绑定构成的路由表，Exchange和Queue的绑定可以是多对多的关系
  • Connection：网络连接，比如一个TCP连接
  • Channel：信道，多路复用连接中的一条独立的双向数据流通道，信道是建立在真实的TCP连接内的虚拟连接，AMQP命令都是通过信道发出去的，不管是发布消息，订阅队列还是接收消息，信道用于复用一条TCP连接，避免频繁TCP建立维护和销毁的开销
  • Consumer：消息的消费者，表示一个从消息队列中取得消息的客户端应用程序
  • Virtual Host：虚拟主机，表示一批交换器，消息队列和相关对象，是共享相同身份认证和加密环境的独立服务器域，每个vhost本质上就是一个mini版的RabbitMQ服务器，拥有自己的队列，交换器，绑定和权限机制，vhost是AMQP概念的基础，必须在连接时指定，RabbitMQ默认的vhost是/
  • Broker：表示消息队列服务器实体

运行机制

AMQP消息路由机制：关键是Exchange和Binding，Exchange四种类型如下：

  • Direct：消息中路由键（routing key）如果和Binding中的binding key一致，交换器就将消息发到对应的队列中，它是完全匹配，单播的模式
  • Fanout：每个发到fanout类型交换器的消息都会分到所有绑定的队列上去，它不处理路由键，只是简单将队列绑定到交换器上，它的转发速度最快，类似于子网广播
  • topic：通过模式匹配分配消息的路由键属性，将路由键和某个模式进行匹配，此时队列需要绑定到一个模式上。它将路由键和绑定键的字符串切分成单词，这些单词之间用点隔开，同样识别两个通配符： ‘#’ 和 ‘#’，#匹配0个或多个单词，*只匹配一个单词
  • headers：基本不用了，所以可以不去了解

配置 缺省值 说明
name 配置这个属性的意义在于，如果存在多个数据源，监控的时候可以通过名字来区分开来。如果没有配置，将会生成一个名字，格式是：”DataSource-“ + System.identityHashCode(this). 另外配置此属性至少在1.0.5版本中是不起作用的，强行设置name会出错。
url 连接数据库的url
username 连接数据库的用户名
password 连接数据库的密码。如果你不希望密码直接写在配置文件中，可以使用ConfigFilter
driverClassName 根据url自动识别 这一项可配可不配，如果不配置druid会根据url自动识别dbType，然后选择相应的driverClassName
initialSize 0 初始化时建立物理连接的个数。初始化发生在显示调用init方法，或者第一次getConnection时
maxActive 8 最大连接池数量
maxIdle 8 已经不再使用，配置了也没效果
minIdle 最小连接池数量
maxWait 获取连接时最大等待时间，单位毫秒。配置了maxWait之后，缺省启用公平锁，并发效率会有所下降，如果需要可以通过配置useUnfairLock属性为true使用非公平锁。
poolPreparedStatements false 是否缓存preparedStatement，也就是PSCache。PSCache对支持游标的数据库性能提升巨大，比如说oracle。在mysql下建议关闭。
maxPoolPreparedStatementPerConnectionSize -1 要启用PSCache，必须配置大于0，当大于0时，poolPreparedStatements自动触发修改为true。在Druid中，不会存在Oracle下PSCache占用内存过多的问题，可以把这个数值配置大一些，比如说100
validationQuery 用来检测连接是否有效的sql，要求是一个查询语句，常用select ‘x’。如果validationQuery为null，testOnBorrow、testOnReturn、testWhileIdle都不会起作用。
validationQueryTimeout 单位：秒，检测连接是否有效的超时时间。底层调用jdbc Statement对象的void setQueryTimeout(int seconds)方法
testOnBorrow true 申请连接时执行validationQuery检测连接是否有效，做了这个配置会降低性能。
testOnReturn false 归还连接时执行validationQuery检测连接是否有效，做了这个配置会降低性能。
testWhileIdle false 建议配置为true，不影响性能，并且保证安全性。申请连接的时候检测，如果空闲时间大于timeBetweenEvictionRunsMillis，执行validationQuery检测连接是否有效。
keepAlive false（1.0.28） 连接池中的minIdle数量以内的连接，空闲时间超过minEvictableIdleTimeMillis，则会执行keepAlive操作。
timeBetweenEvictionRunsMillis 1分钟（1.0.14） 有两个含义： 1) Destroy线程会检测连接的间隔时间，如果连接空闲时间大于等于minEvictableIdleTimeMillis则关闭物理连接。 2) testWhileIdle的判断依据，详细看testWhileIdle属性的说明
numTestsPerEvictionRun 30分钟（1.0.14） 不再使用，一个DruidDataSource只支持一个EvictionRun
minEvictableIdleTimeMillis 连接保持空闲而不被驱逐的最小时间
connectionInitSqls 物理连接初始化的时候执行的sql
exceptionSorter 根据dbType自动识别 当数据库抛出一些不可恢复的异常时，抛弃连接
filters 属性类型是字符串，通过别名的方式配置扩展插件，常用的插件有： 监控统计用的filter:stat 日志用的filter:slf4j 防御sql注入的filter:wall
proxyFilters 类型是List<com.alibaba.druid.filter.Filter>，如果同时配置了filters和proxyFilters，是组合关系，并非替换关系

SpringBoot05-Docker

Docker是一个开源的应用容器引擎

Docker支持将软件编译成一个镜像，然后在镜像中做好各种软件的配置，将镜像发布出去，其他使用者（安装了Docker）可以直接使用这个镜像，就免除了安装和配置过程。运行中的镜像称为容器（容器启动很快）

Docker核心概念：

Docker主机（Host）：操作系统上安装了Docker程序的机器
Docker客户端（Client）：连接Docker主机进行操作，通过命令行或其他工具使用Docker（和MySQL，Redis类似）
Docker仓库（Registry）：用来保存各种打包好的软件镜像
Dokcer镜像：用于创建Docker容器的模板
Docker容器：容器是独立运行的一个或一组应用，镜像启动后的实例就是一个容器

使用docker步骤：

  1. 安装docker
  2. 去docker仓库找到这个软件对应的镜像
  3. 使用docker运行这个镜像，生成一个docker容器
  4. 对容器启动停止就是对软件启动停止

CentOS使用Dokcer

Docker官方仓库文档

对应软件的docker使用官方文档都在仓库里有说明

  • 安装docker： yum install docker
  • 启动docker： systemctl start docker
  • docker开机启动： systemctl enable docker
  • 停止docker： systemctl stop docker

镜像操作

  • 检索： docker search 关键词 docker hub上去检索镜像的详细信息
  • 拉取： docker pull 关键词:tag，如果不指定标签tag，则默认使用latest
  • 列表： docker images 查看所有本地镜像
  • 删除： docker rmi image-id 删除指定本地镜像

容器操作

  • 运行： docker run --name container-name -d image-name:tag 如： docker run --name mytomcat -d tomcat
  • 列表： docker ps 查看正在运行的容器，-a查看所有容器
  • 停止： docker stop container-id(or name) 如： docker stop mytomcat
  • 启动： docker start container-id(or name)，可以先通过 docker ps -a 查看id或name
  • 删除： docker rm container-id(or name)
  • 端口映射： 启动的时候设置端口映射 docker run --name container-name -d -p hostport:containerport imagename:tag 将主机端口映射到容器内部端口 -d：后台运行；-p：主机端口映射到容器端口
  • 容器日志： docker logs name(id)

以MySQL为例

使用docker，需要参照仓库里的官方文档，如果没有参照，可能会报如下错误
安装MySQL后，启动，ps -a命令发现MySQL没有成功启动，使用日志命令查看，发现需要指定密码参数
则使用 docker run --name some-mysql -e MYSQL_ROOT_PASSWORD=my-secret-pw -d mysql:tag（官方文档有说明）
需要注意做端口映射： docker run -p 3306:3306 --name some-mysql -e MYSQL_ROOT_PASSWORD=my-secret-pw -d mysql:tag
添加开机自启动： docker update --restart=always xxx
其他配置（比如配置文件映射等）可以参考docker的MySQL官方文档
docker内执行mysql命令：docker exec -it xxx bash，然后mysql -u root -p

SpringMVC自动配置原理

  • 配置了ContentNegotiatingViewResolver and BeanNameViewResolver beans.，即ViewResolver视图解析器，视图解析器根据方法的返回值得到视图对象（View），视图对象决定如何渲染（转发或重定向）
  • 配置了静态资源文件夹路径，webjars
  • 静态首页 index.html 的支持
  • 图标 Favicon 的支持
  • 自动注册了 Converter , Formatter
    • Converter：转换器，类型转换
    • Formatter：格式化器：比如日期
    • 自己添加的格式化器转换器，只需要放在容器中即可
  • 支持HttpMessageConverters，Spring MVC用来转换Http请求和响应
  • MessageCodesResolver定义错误代码生成规则
  • ConfigurableWebBindingInitializer初始化WEB数据绑定器，WEB数据绑定器就是把请求数据封装到Java Bean

If you wish to provide custom instances of RequestMappingHandlerMapping, RequestMappingHandlerAdapter, or ExceptionHandlerExceptionResolver, you can declare a WebMvcRegistrationsAdapter instance to provide such components.
If you want to take complete control of Spring MVC, you can add your own @Configuration annotated with @EnableWebMvc.

软件工程期末复习

概论

软件是计算机系统的逻辑部件，固有特点是缺乏可见性，管理和控制软件开发过程相当困难

软件危机：计算机软件开发与维护过程中所遇到的一系列严重问题称为软件危机

软件工程：是指导计算机软件开发和维护的一门工程学科，该学科目的是生产出能按期交付的，在预算范围内的，满足用户需求的，质量合格的软件产品

软件工程方法学：把在软件生命周期全过程中使用的一整套技术方法的集合称为方法学，也称为范型，包含3要素：方法，工具和过程，现在使用最广泛的有传统方法学和面向对象方法学
方法：完成软件开发各项任务的技术方法
工具：运用方法而提供的自动的或半自动的软件工程支撑环境
过程：为了获得高质量软件所需要完成的一系列任务的框架，规定了各项任务的工作步骤

软件生命周期：软件从提出到最终被淘汰的这个存在期，由软件定义（问题定义，可行性研究，需求分析），软件开发（总体设计，详细设计，编码和单元测试，综合测试）和运行维护3个时期组成

软件过程：为了获得高质量软件所需要完成的一系列任务的框架，它规定了完成各项任务的工作步骤，生命周期模型可以概括描述软件过程

问题定义阶段：确定用户建立什么样的软件系统，软件叫什么名等，问题定义报告的内容包括：软件项目标题，软件目标，软件用户对象，软件规模

可行性研究

可行性研究阶段目的：用最小代价在尽可能短的时间内研究并确定客户提出的问题是否有行得通的解决方法（技术可行性，经济可行性，操作可行性，社会可行性）

数据流图

箭头上标识数据流具体是什么（比如存款单，存款信息）
变换数据的处理标识对这个数据具体做什么，存储标识存储的是什么
处理可以表示程序处理等，数据存储可以表示文件，数据库等

数据字典：对数据流图中包含的所有元素的定义的集合（由数据流，数据元素，数据存储，处理组成）

  • =：等价于或定义为
  • +：和（连接两个分量）
  • []：或，用|分隔
  • {}：重复，{}里的元素可以任意个重复，也可以1{A}5标识上下限
  • ()：可选（可有可无）

可行性研究阶段，数据流图和数据字典共同构成系统的逻辑模型

需求分析

需求分析：确定系统必须完成哪些工作

与用户沟通获取需求的方法：访谈，面向数据流自顶向下求精，简易的应用规格说明技术，快速建立软件原型

状态转换图：

状态：任何可以被观察到的系统行为模式，状态图只有1个初态，可以有任意终态
事件：在某个特定时刻发生的事情，引起系统状态转换

描绘单程生命期时，需要标明初始状态和最终状态，其他则不需要？
初态用实心圆表示，终态用一对同心圆（内圆实心，见上图）表示，中间状态用圆角矩形表示
状态名必须有，状态变量的名字和值是可选的，活动表也是可选的

活动表语法：事件名（参数表）/动作表达式
事件名可以是任何时间的名称，常用的三个标准事件：entry，exit和do。entry事件指定进入该状态的动作，exit事件指定退出该状态的动作，do事件指定该状态下的动作
箭头线上标出触发转换的事件表达式，若未指明则在源状态内部活动执行完后自动触发转换

其他图形工具：层次方框图，Warnier图，IPO图

验证软件需求：从以下4方面验证需求的正确性：一致性，完整性，现实性，有效性

形式化说明技术

形式化方法是描述系统性质的基于数学的技术

非形式化方法：自然语言描述
半形式化方法：数据流图或实体-联系图
形式化方法：基于数学技术描述

有穷状态机：有穷非空状态集J；有穷非空输入集K；转换函数T；初始态S；终态集F

Z语言：给定的集合，数据类型及常数；状态定义；初始状态；操作

总体设计

设想供选择的方案，选择合理的方案，推荐最佳方案，功能分解，设计软件结构，数据库设计，制定测试计划，书写文档，审查和复审

设计原理：模块化，抽象，逐步求精，信息隐蔽和局部化，模块独立（由耦合和内聚定性标准度量）

耦合：软件结构内不同模块彼此间相互依赖的紧密程度，分为数据耦合，控制耦合，公用耦合，内容耦合（尽量使用数据耦合，少用控制耦合，限制公用耦合，完全不用内容耦合）
数据耦合：两个模块彼此间通过参数交换信息，而且交换的信息仅仅是数据
控制耦合：如果传递的信息中有控制信息，则为控制耦合
公用耦合：两个或多个模块通过一个公共区相互作用时的耦合，公共区指全程数据区，共享通信区，内存公共覆盖区，文件，物理设备等
内容耦合：一个模块与另一个模块的内容直接发生联系

内聚：一个模块内部各个元素彼此结合的紧密程度（功能内聚，顺序内聚，通信内聚，过程内聚，时间内聚，逻辑内聚，偶然内聚），力求做到高内聚（功能和顺序），尽量少用中内聚（通信和过程），不用低内聚（偶然）
功能内聚：一个模块所有元素完成且仅完成一个功能，称为功能内聚
顺序内聚：一个模块内处理元素和同一个功能密切相关，且这些处理元素必须顺序执行
通信内聚：一个模块所有处理元素都是用同一个输入数据和产生同一个输出数据
过程内聚：一个模块内处理元素是相关的，且必须以特定次序执行
时间内聚：包含的任务必须同一段时间内执行
逻辑内聚：模块完成的任务在逻辑上属于相同或相似一类

描绘软件结构的图形工具：HIPO图，层次图

面向数据流的设计方法：把信息流映射成软件结构，信息流的类型决定了映射的方法
数据流分为变换流和事务流

详细设计

目标：确定如何具体实现所要求的系统。它决定了最终程序代码的质量

结构程序设计：采用自顶向下逐步求精的设计方法和单入口单出口的控制结构
正规程序：一个入口线和一个出口线，对每个结点都有一条从入口线到出口线的通路通过

非结构化程序到结构化程序的转换看目录下的图1-7

人机界面设计：问题：系统响应时间，用户帮助，出错信息处理，命令交互
人机界面设计指南：一般交互指南，信息显示指南，数据输入指南

过程设计的工具：程序流程图，盒图，PAD图，判定表，判定树，过程设计语言
面向数据结构的设计方法：Jackson图，改进Jackson图，Jackson方法

程序复杂程度的定量度量：McCabe方法，Halstead方法

实现

编码和测试统称为实现
编码：把软件设计结果翻译成程序
测试：检测程序并改正错误的过程

软件测试的目标：1.为了发现程序中的错误而执行程序的过程；2.好的测试方案极可能发现迄今为止尚未发现的错误的测试方案；3.成功的测试是发现了至今为止尚未发现的错误的测试

黑盒测试：如果已经知道软件应该具有的功能，可以通过测试来检验是否每个功能都能正常使用，这种测试称为黑盒测试，也叫功能测试

白盒测试：也叫结构测试，如果知道软件内部工作过程，可以通过测试来检验软件内部动作是否按照规格说明书的规定正常进行

测试步骤：模块测试 -> 子系统测试 -> 系统测试 -> 验收测试（用户实际数据） -> 平行运行（与旧系统比较）

单元测试：集中检测软件设计的最小单元-模块；单元测试主要使用白盒测试技术；首先编译检查语法错误，然后用详细设计模块说明为指南，对重要的控制路径进行测试，以便发现模块内部的错误
测试重点：模块接口，局部数据结构，重要的执行路径，出错处理通路，边界条件

代码审查：审查小组：组长，程序设计者，程序编写者，程序测试者

计算机测试：模块并不是独立的程序，因此必须为每个单元测试开发驱动软件和存根软件；驱动软件接收数据并传送给被测试的模块并打印结果，存根软件代替被测试的模块所调用的模块

集成测试：是组装软件的系统化技术，它将经过单元测试的模块联系在一起进行测试
两种方法
非渐增式测试方法：先分别测试每个模块，再把所有模块按设计要求放在一起结合成所要的程序
渐增式测试方法：每次增加一个待测试模块，把它同已经测试好的那些模块结合起来进行测试，反复进行直到完成所有模块测试的方法
渐增式分为自顶向下和自底向上集成
自顶向下集成：需要存根程序，不需要驱动程序，从主程序开始沿软件的控制层次向下移动逐渐把各个模块结合起来，分为深度优先和宽度优先策略
自底向上集成：从软件最底层模块开始进行组装和测试，需要驱动程序，不需要存根程序
回归测试：重新执行已经做过的部分测试，用于保证由于调试或其他原因引起的程序变化，不会导致额外错误的测试活动

确认测试：目标是验证软件的有效性；一般使用黑盒测试法
软件配置复查：保证软件配置的所有成分都齐全，质量符合要求，文档与程序完全一致，而且编好目录
Alpha测试：用户在开发者的场所进行测试，并且在开发者的指导下进行，测试在受控环境中，开发者记录发现的错误和问题
Beta测试：用户在一个或多个客户场所进行测试，不受开发者控制，测试者记录发现的问题和错误，定期把问题报告发送给开发者

白盒测试技术：
逻辑覆盖：有如下几种覆盖
语句覆盖：每个语句至少执行一次
判定覆盖：每个判定的每种可能的结果至少执行一次（每个判定分支至少通过一次）

条件覆盖：判定表达式中每个条件都取到各种可能的结果

判定/条件覆盖
条件组合覆盖：每个判定表达式中条件的各种可能组合都至少出现一次
点覆盖：选组足够多的测试用例，使得程序执行路径至少经过程序图中每个节点一次
边覆盖：选取足够多的测试用例，使得程序执行路径至少经过程序图中每条边一次
路径覆盖：选取足够多的测试用例，使得程序的每条可能路径都至少执行一次

黑盒测试技术
等价划分：将所有可能的输入数据划分为若干个等价类，就可以假定用每个等价类中的代表值作为测试用例进行测试时，等价于用该类所有值进行测试
边界值分析：使用正好等于，正好大于，正好小于边界值的数据进行测试，发现程序错误的概率较大，属于黑盒测试
实用测试策略：任何情况下豆使用边界值分析；必要时用等价划分法补充；必要时再用错误推测法补充；根据程序可靠性要求采用不同逻辑覆盖标准

调试：在测试发现错误之后排除错误的过程
调试途径：
蛮干法，回溯法，原因排除法

软件可靠性：程序在给定的时间间隔内，按照规格说明书的规定成功运行的概率
对故障可修复系统，应同时使用可靠性和可用性衡量
软件可用性：程序在给定的时间点，按照规格说明书的规定，成功运行的概率

平均维修时间MTTR是修复一个故障平均需要用的时间，取决于维护人员的技术水平和对系统熟悉程度。
平均无故障时间MTTF是系统按照规格说明书规定成功地运行的平均时间，取决于系统中潜伏的错误数量

维护

维护软件正常运行，不断改进软件的性能和质量，为软件进一步推广应用和更新替换做积极工作

软件交付使用的方式：直接，并行，逐步

软件维护的类型：改正性维护（诊断和改正错误），适应性维护，完善性维护（新功能、修改现有功能），预防性维护

软件可维护性：由5个因素决定：可理解性，可测试性，可修改性，可移植性，可重用性

面向对象方法学引论

基本原则：尽可能模拟人类习惯的思维方式，使开发软件的方法与过程尽可能接近人类认识世界解决问题的方法与过程

面向对象方法四个要点：认为客观世界由对象组成；把所有对象都划分为各种对象类；把若干对象类组成一个层次结构的系统；对象彼此间仅通过传递消息互相联系

对象特点：数据为中心；主动；实现数据封装；本质上具有并行性；模块独立性好

类：具有相同数据和相同操作的一组相似对象的定义
消息：用来请求对象执行某个处理或回答的某些信息的要求，既可以是数据流，也可以是控制流；组成：接收信息的对象；信息选择符（消息名）；0个或多个变元（参数），如：MyCircle.Show(GREEN)
多态性：子类对象可以像父类对象那样使用，同样的消息既可以发送给父类对象，也可以发送给子类对象
重载：函数重载和运算符重载

对象模型：描述系统的数据结构
动态模型：描述系统的控制结构

功能模型
用例图包含的模型元素有系统，行为者，用例和用例间的关系
用例：行为者感受到的一个完整的功能，它是类，代表一类功能
行为者：与系统交互的人或其他系统，它代表一种角色
用例之间的关系：

  • 扩展关系：向用例添加一些动作后构成另一个用例，它们之间构成扩展关系
  • 使用关系：一个用例使用另一个用例，它们之间构成使用关系

注意！用例是一个类，它代表一类功能，而不是使用该功能的某个具体的实例；行为者代表一种角色，而不是具体某个人或物

面向对象设计

软件重用：3个层次：知识重用，方法和标准重用，软件成分重用
软件成分重用级别：代码重用，设计结果重用，分析结果重用

软件项目质量

软件质量：与软件产品满足规定的和隐含的需求的能力有关的特征或特性的全体

三个方面考虑软件质量：功能与性能方面，软件结构方面，开发标准与文档方面

ISO质量度量模型：高层是软件质量需求评价准则（质量特性），中层是质量设计评价准则（质量子特性），底层是质量度量评价准则（度量）

软件工程管理：指对整个软件开发运行过程，包括分析，设计，编码，测试及维护各个阶段的一切活动进行有效的管理和控制，包括开发人员，组织机构，用户，文档资料，计划进度，经费开支等方面的管理
软件工程控制：进度控制，人员控制，经费控制，质量控制
软件工程管理很大程度上是通过对文档资料的管理来实现的

软件产权：软件产品的特殊性是：软件开发工作量大、周期长、投资高，然而软件复制却相当容易，而且复制费用很低
相关法律：版权法，专利法，商标法，商业秘密法

版权法主要保护对象是文字作品。计算机软件中的文档资料大部分是文字作品，源程序中的注释部分也是文字作品。
如果将代码看作是特殊的文字，整个程序就是文字作品了。所以，版权法保护计算机软件
版权法要求受保护的软件产品具有独创性

对软件产品的保护，专利法比版权法要求严格。它要求受保护的软件具有三个特性：新颖性、技术先进性、实用性

软件著作权规定受保护的软件必须满足两个基本条件：

  • 必须是开发者独立开发的，即具有原创性
  • 在某种有形物体上（纸，磁带）
