SpringMVC Auto-configuration

Spring Boot provides auto-configuration for Spring MVC that works well with most applications.

The auto-configuration adds the following features on top of Spring’s defaults:

Inclusion of ContentNegotiatingViewResolver and BeanNameViewResolver beans.
Support for serving static resources, including support for WebJars (covered later in this document)).
Automatic registration of Converter, GenericConverter, and Formatter beans.
Support for HttpMessageConverters (covered later in this document).
Automatic registration of MessageCodesResolver (covered later in this document).
Static index.html support.
Custom Favicon support (covered later in this document).
Automatic use of a ConfigurableWebBindingInitializer bean (covered later in this document).

SpringMVC自动配置原理

  • 配置了ContentNegotiatingViewResolver and BeanNameViewResolver beans.，即ViewResolver视图解析器，视图解析器根据方法的返回值得到视图对象（View），视图对象决定如何渲染（转发或重定向）
  • 配置了静态资源文件夹路径，webjars
  • 静态首页 index.html 的支持
  • 图标 Favicon 的支持
  • 自动注册了 Converter , Formatter
    • Converter：转换器，类型转换
    • Formatter：格式化器：比如日期
    • 自己添加的格式化器转换器，只需要放在容器中即可
  • 支持HttpMessageConverters，Spring MVC用来转换Http请求和响应
  • MessageCodesResolver定义错误代码生成规则
  • ConfigurableWebBindingInitializer初始化WEB数据绑定器，WEB数据绑定器就是把请求数据封装到Java Bean

配置SpringMVC

If you want to keep Spring Boot MVC features and you want to add additional MVC configuration (interceptors, formatters, view controllers, and other features), you can add your own @Configuration class of type WebMvcConfigurer but without @EnableWebMvc. If you wish to provide custom instances of RequestMappingHandlerMapping, RequestMappingHandlerAdapter, or ExceptionHandlerExceptionResolver, you can declare a WebMvcRegistrationsAdapter instance to provide such components.
If you want to take complete control of Spring MVC, you can add your own @Configuration annotated with @EnableWebMvc.

（如果只想要在默认配置上添加一些例如拦截器等MVC功能，则只需要在配置类实现 WebMvcConfigurer 接口并实现对应方法即可，而如果要完全接管MVC，全部自己配置，则需要再添加 @EnableWebMvc 注解，这个注解引入了 DelegatingWebMvcConfiguration 配置类来做默认实现

原理

注：可能我的理解有错，所以这里可能会出现错误

MVC通过 WebMvcAutoConfiguration 来实现自动配置MVC，它配置了suffix，prefix等，它还有一个注解 @ConditionalOnMissingBean({WebMvcConfigurationSupport.class}) 当容器中缺失 WebMvcConfigurationSupport 这个组件的时候，自动配置类就起作用
当使用 @EnableWebMvc 注解全面接管MVC的时候，它引入了 DelegatingWebMvcConfiguration 配置类，而这个类就是 WebMvcConfigurationSupport 的子类，所以实现了该接口的配置会导致自动配置类失效，这个类是通过 WebMvcConfigurerComposite 类起作用，它内部有个域 List<WebMvcConfigurer>，调用相应配置方法时，就迭代这个List，让所有MVC配置类都生效（因此我们的MVC配置类需要实现 WebMvcConfigurer 接口，才能被其迭代到并调用我们配置MVC配置类的MVC配置，比如添加拦截器等）

自动配置类中有一个内部配置类

继承自 DelegatingWebMvcConfiguration ，把我们自己写的WebMvcConfigurer也顺便实现了

错误处理机制

默认错误处理机制

如果是浏览器访问，返回一个默认错误页面

如果是其他客户端访问，则返回一个JSON数据，里面包括状态码，请求路径，错误信息等

他们都是通过 ErrorMvcAutoConfiguration.java 自动配置的（自动配置过程略，但过程涉及我们定制错误响应，目前我觉得知道怎么用就行）

定制错误响应

定制错误页面

  1. 有模板引擎的情况下，创建 模板引擎目录/error/状态码.html 作为对应状态码的错误页面响应
    或者使用 模板引擎目录/error/4xx.html，模板引擎目录/error/5xx.html 来匹配所有以4或5开头的错误码的页面（如果还额外添加了具体的比如404.html，则优先使用404.html）
    页面能获取的信息：（在 DefaultErrorAttributes.java ）
    - timestamp：时间戳
- status：状态码
- error：错误提示
- exception：异常对象
- message：异常消息
- errors：JSR303数据校验的错误都在这
- 在thymeleaf中，可以使用 `[[${status}]]` 来获取对应信息
  2. 没有模板引擎（模板引擎目录下没有对应错误页面），则在 静态资源文件夹/error/状态码.html
  3. 以上都没有，则来到Spring Boot默认的错误页面

定制错误JSON数据

  1. 编写自己的测试异常
  1. Controller编写抛出这个异常
  1. 全局配置开启 server.error.include-exception=true （这样才能显示message信息）
  2. 编写异常处理器

这样，当接收到 UserNotFoundException 的时候，就会交给这个异常处理器处理，这个异常处理器是 @ResponseBody 注解，会把对应map返回成一个json格式的数据给客户端（包括网页和其他客户端）（如果要自适应，则重写 ErrorAttributes ）
